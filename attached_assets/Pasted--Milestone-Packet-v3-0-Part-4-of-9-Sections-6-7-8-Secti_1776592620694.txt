Section 7 — Priority 5: Framework Inference Subsystem
Carries forward from v2.0 Section 5 with integration to canonicalization (Section 5) and semantic detection (Section 4).
7.1 Three tiers of professor input
Tier
Professor Input
System Behavior
A
Two or more frameworks
Track exactly those. No inference. All provenance = explicit.
B
Exactly one framework
Track the explicit one. Propose up to 3 complementary frameworks tagged inferred_from_anchor. Professor opts in per framework.
C
None
Propose up to 3 frameworks inferred from case metadata and pedagogical intent. Tagged inferred_from_context. Professor opts in per framework.
7.2 Hard constraints
Total inferred frameworks never exceeds 3, regardless of tier.
Every inferred framework carries a human-readable inference_reason.
Opt-in mandatory. accepted_by_professor = false → engine ignores entirely.
Professor may remove or replace any framework before the case goes live.
Once the scenario has any session in status active or completed, framework configuration is locked.
7.3 Implementation
Create server/agents/frameworkInference.ts. Called once per scenario, at case-creation time, after canonical generation.
export interface InferenceInput {
caseContext: string;
domain: string;
industry?: string;
language: "es" | "en";
anchorFramework?: CaseFramework;      // Tier B
pedagogicalIntent?: PedagogicalIntent; // from Section 8
courseTitle?: string;
}
export async function inferFrameworks(input: InferenceInput): Promise<{
suggestions: Array<{
canonicalId: string;               // resolved from registry when possible
name: string;
domainKeywords: string[];
coreConcepts: string[];
conceptualDescription: string;
recognitionSignals: string[];
signalPattern?: CaseFramework["signalPattern"];
provenance: "inferred_from_anchor" | "inferred_from_context";
inference_reason: string;
}>;
}>
Rules
If anchorFramework present: prompt for 2–3 complementary frameworks. Resolve each against the registry. For registry-known frameworks, pull all semantic fields from registry. Tag inferred_from_anchor.
If absent: prompt for 3 frameworks most likely relevant to case + pedagogical intent. Same registry resolution. Tag inferred_from_context.
Use gpt-4o-mini, responseFormat: json, maxTokens: 768.
Post-process through frameworkKeywordSanitizer and the language-consistency assertion from Section 3.3.
Truncate to 3 if model over-produces.
Deduplicate against explicit frameworks by canonicalId.

Section 8 — Priority 6: The Pedagogical Intent Layer
The framework field added between T-001 and T-002 is a good first step but is not enough. The professor still cannot say what they want students to TEST versus what they want to EXPLORE. Pedagogical Intent is the structured input channel that closes this gap. Carries forward from v2.0 Section 6.
8.1 The PedagogicalIntent object
Add to shared/schema.ts:
export interface PedagogicalIntent {
// Required. The one-sentence teaching goal.
teachingGoal: string;
// Optional. Canonical framework IDs the case should exercise.
// Resolved via the registry (Section 5). Raw names also accepted
// for custom frameworks.
targetFrameworks: Array<{
canonicalId: string | null;          // null = custom
name: string;
}>;
// Optional. Which of C1..C5 to emphasize in case design
// and in runtime calibration.
targetCompetencies: Array<"C1" | "C2" | "C3" | "C4" | "C5">;
// Optional. Per-decision dimension assignment.
decisionDimensions?: Array<{
decisionNumber: number;
primaryDimension: AcademicDimension;
secondaryDimension?: AcademicDimension;
}>;
// Optional.
courseContext?: string;
reasoningConstraint?: string;           // e.g. "time pressure"
}
export type AcademicDimension =
| "analytical" | "strategic" | "stakeholder"
| "ethical" | "tradeoff";
8.2 Storage
Add pedagogical_intent JSON column on the scenarios table. Include in every scenario read. Expose via GET /api/scenarios/:id. Editable via PATCH /api/scenarios/:id/pedagogical-intent but only when the scenario has zero sessions.
8.3 UI — Teaching Intent page
Add a new step in the scenario creation wizard, BEFORE case generation. Fields:
Teaching goal (required). Single text input. Placeholder: "Students should apply stakeholder analysis to operational decisions under time pressure."
Target frameworks (optional, 0–4). Reuse the FrameworkEditor at this stage. Each entry runs through the registry resolver.
Target competencies (optional, 0–5). Checkbox list of C1–C5 with human-readable labels.
Per-decision dimension (optional, advanced). Expandable per-decision section.
Course context (optional). Text input.
Reasoning constraint (optional). Text input.
8.4 Required vs optional
Only teachingGoal is required. Fallbacks for everything else:
targetFrameworks empty → Framework Inference Subsystem runs at Tier C using pedagogicalIntent as input.
targetCompetencies empty → all five weighted equally.
decisionDimensions empty → Section 10 rules auto-assign.

## B2. Framework Detection Engine Pass

**What it does:** On every FR turn (not MCQ), after the 5 signals are extracted, run a pass that classifies each professor-specified framework as `explicit`, `implicit`, or `not_evidenced` based on the student's response + extracted signals.

**New agent file:** `server/agents/frameworkDetector.ts`.

**Detection rules (implement exactly):**
- **Explicit** — Student's response contains the framework name, a named theorist, or any domain keyword (case-insensitive regex match against the framework's `domainKeywords` array).
- **Implicit** — All of:
  - Framework's `signalPattern.requiredSignals` are all present at or above `signalPattern.minQuality` in the turn's extracted signals.
  - At least one non-technical domain keyword or a conceptually-related word appears in the student's response OR the student's extracted signals reference the domain (e.g., stakeholder awareness mentions a geographic/cultural element for a cultural-distance framework).
- **Not evidenced** — Neither condition met.

**For MCQ (Turn 1):** Skip framework detection. MCQ responses don't carry enough signal content. Return an empty `framework_detections` array.

**Session state addition** in `shared/schema.ts`:
```typescript
export interface FrameworkDetection {
  framework_id: string;
  framework_name: string;
  level: "explicit" | "implicit" | "not_evidenced";
  evidence: string;  // 1 sentence describing what in the student's input produced this
}
```

Add to the per-turn schema (where `decisionEvidenceLogs` lives):
```typescript
framework_detections?: FrameworkDetection[];
```

**Evidence sentence rules** (enforce via post-generation checks):
- Maximum 1 sentence.
- References what the student actually wrote (at least one word/phrase from `student_input`).
- No evaluation language ("good", "bad", "correct", "incorrect", "should have").
- Uses scenario language.

**Pipeline integration** in `server/agents/director.ts`: After signal extraction completes and before the parallel S3+S4 block, add:
```typescript
const frameworkDetections = scenario.initialState?.frameworks?.length
  ? await detectFrameworks(
      contextWithRDS,
      evidenceLog.signals_detected,
      scenario.initialState.frameworks
    )
  : [];
// Store frameworkDetections in the turn record alongside decisionEvidenceLog
```

**Done looks like:**
- Every FR turn produces a `framework_detections` array with one entry per professor-specified framework.
- Each entry has a level and a grounded 1-sentence evidence string.
- MCQ and scenarios without frameworks produce `[]`.
- Scenario language respected in evidence strings.

## B4. Session-Level Dashboard Summary

**What it does:** When the final turn completes, compute and store aggregation fields on the session so the dashboard doesn't recompute on every load.

**Schema addition** in `shared/schema.ts`:
```typescript
export interface DashboardSummary {
  session_headline: string;           // 1–2 sentence AI-generated session summary
  signal_averages: {                  // pre-computed for radar chart (0–3 scale, 1 decimal)
    analytical: number;
    strategic: number;
    tradeoff: number;
    stakeholder: number;
    ethical: number;
  };
  framework_summary: Array<{
    framework_id: string;
    best_level: "explicit" | "implicit" | "not_evidenced";
    turn_of_best_application: number | null;
  }>;
}
```

Add to `SimulationState`:
```typescript
dashboard_summary?: DashboardSummary;
```

**Where to compute:** In `server/agents/director.ts`, at the end of `processStudentTurn` when `decisionsComplete === true`, generate and persist this.

**Session headline generation rules (inline — enforce exactly):**
- 1–2 sentences maximum.
- References the student's reasoning arc (which turns were Surface/Engaged/Integrated).
- Identifies the single most significant gap OR strength (not both).
- Written as observation, not evaluation.
- Uses scenario language.
- May note if pattern differs from class average (optional).
- Prohibited: "correct/incorrect", "good/bad decision", "well done", "optimal", "ideal", "should have", "you should", "best", "unfortunately", "fortunately", "sadly", "surprisingly", any exclamation marks.
- Prohibited dashboard-specific: "weak student", "strong performer", any comparison by student name.

**Signal averages computation:**
```typescript
// For each of the 5 signals:
const average = signalValues.reduce((sum, v) => sum + v, 0) / signalValues.length;
// Where signalValues = turn-level signal quality (0-3) across all completed turns
// Round to 1 decimal.
```

**Framework summary computation:** Per framework, find the turn where `level` was highest (priority: explicit > implicit > not_evidenced) and record `turn_of_best_application`. If never applied anywhere: `best_level: "not_evidenced"`, `turn_of_best_application: null`.

**Done looks like:**
- Completing a session triggers generation of `dashboard_summary`.
- Headline in scenario language, ≤2 sentences, no prohibited terms.
- Signal averages and framework summary numerically correct (verify manually from session turn data).

## B5. Per-Turn Debrief Question Generation

**What it does:** After each turn completes, generate a professor-facing debrief question targeting this specific student's most significant signal gap at that turn.

**Schema addition** — add to each turn's record:
```typescript
dashboard_debrief_question?: string;
```

**Generation rules (enforce exactly):**
1. Must reference something the student actually wrote at this turn (paraphrase or short quote from `student_input`).
2. Must target the lowest-scoring signal at this turn (or the signal that dropped most from prior turn, if prior exists).
3. Phrased as a genuine open question — no leading questions, no implied "right answer".
4. Must not contain any prohibited language (same list as B4 session headline + no "should have", "would have been better", "missed the opportunity to").
5. Must not imply the student was wrong or made a mistake.
6. Maximum 1 sentence.
7. Uses scenario language.

**Fallback template** when 3 retries all fail quality gate:
```
// Spanish:
"En el Turno [N], tú [paráfrasis de la intención] — ¿qué estabas sopesando que te llevó a esa dirección?"
// English:
"In Turn [N], you [paraphrase of intent signal] — what were you weighing that led you to that direction?"
```

**Quality gate logic:**
1. Generate once.
2. Scan for prohibited patterns.
3. If fail: retry with violations appended to prompt. Retry max 2 times.
4. If still failing after 3 total attempts: use fallback template.

**Where to generate:** In `server/agents/director.ts`, after the causal explainer finishes. Can run in parallel with `dashboard_reasoning_link` generation if desired, or as a separate method in `causalExplainer.ts` called right after the main explanation pass.

**Done looks like:**
- Every turn record has a populated `dashboard_debrief_question`.
- The question references the student's specific words.
- It targets the signal gap.
- No prohibited language.
- Scenario language respected.
- Fallback template only used when 3 retries fail (should be rare).

### C2.1 — Top stats row (4 cards)

**Four white cards in a row:**

| Card | Value source | Description (inline under value) |
|---|---|---|
| Completed | Count of sessions where `session_status === "complete"` | "Number of students who have finished all turns and submitted the reflection" |
| In progress | Count of sessions where `session_status === "active"` | "Number of students who have started but not yet completed the simulation" |
| Biggest drop point | Computed (see below) | "The turn where average reasoning depth dropped most significantly across the class" |
| Applied course theory | Computed (see below) | "How many students showed evidence of applying at least one professor-specified framework" |

**Biggest drop point computation:**
1. For each turn N, compute class average RDS band: `avg = mean(student.turn_N.rds_band)` where Surface=1, Engaged=2, Integrated=3.
2. Compute deltas: `delta[i] = avg[i] - avg[i-1]` for i from 2 to N.
3. The turn with the largest negative delta = biggest drop point.
4. If no negative delta exists → display "—".

**Applied course theory count:**
- Count students who have at least one `framework_detections[].level === "explicit"` OR `"implicit"` across any turn of their completed session.
- Display as "N of M" where M = completed students, N = students with applied theory.
- If no frameworks configured on scenario → display "—".

**API endpoint:** `GET /api/scenarios/:scenarioId/class-stats`
```json
Returns: {
  completed: number,
  inProgress: number,
  biggestDropPoint: { turn: number, delta: number } | null,
  appliedCourseTheory: { n: number, m: number } | null
}
```

### C2.4 — Class Reasoning Patterns (right card)

**Right card in the 2-column row.** Title "Where the class is and where it's heading". Subtitle "The most significant reasoning patterns detected across all completed sessions — described with what the data actually shows, not an evaluation".

**Content:** One block per competency. Show all 5 competencies ranked by demonstration rate, OR show top 2 + bottom 2 (4 blocks total). Recommendation: show 4 blocks (clearest visual).

Each block:
- Competency name (C1 Analytical reasoning / C2 Strategic decision-making / C3 Stakeholder consideration / C4 Ethical reasoning / C5 Tradeoff awareness — use canonical labels)
- Status badge (Transferring / Developing / Not yet evidenced) — same colors as module health
- Progress bar — width = class demonstration rate 0–100%, never displayed as a number. Fill color matches the status badge color.
- 1 sentence AI-generated description

**Class demonstration rate per competency:**
```
For each competency Cx:
  count = 0
  total = 0
  for each student session:
    for each turn:
      total += 1
      if competency_evidence[Cx] in ["demonstrated", "emerging"]:
        count += 1
  rate = count / total  // 0.0 to 1.0
```

**Status thresholds (same as module health):**
- Transferring: rate ≥ 0.60
- Developing: 0.20 ≤ rate < 0.60
- Not yet evidenced: rate < 0.20

**Description generation rules:**
- 1 sentence per competency.
- Describes what was observed. No evaluation.
- Uses scenario language.
- Examples:
  - ✅ "Most students justified decisions with case-specific evidence and followed a logical structure. Most consistent competency across the class."
  - ✅ "Decisions were framed around outcomes, not the people those outcomes affected. Present in one student at one turn across the full class."
  - ❌ "Students need to improve their ethical reasoning." (prescriptive — prohibited)

**API endpoint:** bundled into the `/module-health` endpoint or a new `/class-patterns`.

  - Absent / Negative: text `#791F1F`, bg `#FCEBEB`, accent `#D85A30`

### C4.2 — Empty and edge states

- Class overview with 0 completed sessions: show "No sessions yet" empty state. Don't render charts.
- Class overview with 1 completed session: all sections render correctly with N=1 data.
- No frameworks configured on scenario: module health section shows "No frameworks configured for this case. Add frameworks in the scenario editor to see how they're being applied by students." Hide the framework grid, debrief opener, and expandable deeper view.
- Biggest drop point when depth never dropped: show "—".
- Radar chart with all-zero signal values (student who received all BLOCKs): renders as a single center point, no crash.
- Student in progress: "View session" disabled with opacity 0.35.

### C4.3 — Language support

Every AI-generated dashboard text must respect the scenario's language:
- English scenarios → all dashboard AI text in English
- Spanish scenarios → all dashboard AI text in Spanish
- UI chrome (static labels, buttons, section titles) uses the professor's UI language from the existing `useTranslation()` hook

### C4.4 — Performance

- Aggregation endpoints < 500ms for a class of 30 students.
- `dashboard_summary` computed ONCE on session completion (B4) — stored in session state, never recomputed.
- Module health descriptions, class pattern descriptions, turn annotations: cached for 5 minutes at the scenario level. Cache invalidated when a new session completes or the scenario is edited.
- Per-student modal content: read directly from stored session state on-demand, no cache needed.

### C4.5 — Authorization

- All dashboard endpoints behind professor-role authorization.
- Only the owning professor can access their scenario's dashboard.
- Students NEVER see the dashboard or any of its endpoints.

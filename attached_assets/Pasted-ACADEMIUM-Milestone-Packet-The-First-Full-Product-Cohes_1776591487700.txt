Section 0 — How to Use This Packet
0.1 Why this is v3.0
Three prior packets brought Academium from concept to where it is today. v1.0 specified the product. v1.1 mapped what was broken. v2.0 added the pedagogical architecture. Between v2.0 and this document, the team ran a second full test round (T-002 and T-002A on 2026-04-18) that revealed three things:
Meaningful progress since T-001: the framework entry field now exists, the generated case is visibly theory-shaped, decision continuity across 5 turns is strong, and edit-before-publish is working.
One bug from T-001A has not been fixed: the completion-state sync is still broken. The dashboard still reports 0 completed / 1 in progress when a student has actually completed the session. Language inconsistency is also still present.
A central new problem that no prior packet addressed adequately: framework detection is broken at a fundamental level. Porter's Generic Strategies was clearly applied across all 5 decisions, yet the dashboard marked it as Not Evidenced on every turn. The detector is keyword-anchored even in its implicit path.
This packet supersedes v1.0, v1.1, and v2.0. Engineering and Replit work from this single document only. v3.0 keeps everything from v2.0 that is still correct, updates the inventory, adjusts priorities, and adds two new sections (Semantic Framework Detection and the Cohesion Architecture) that turn Academium into the first version that deserves to be called a full product.
0.2 The milestone
When this packet is implemented and passes its acceptance checklist, Academium will for the first time be cohesive: the engine, the dashboard, the data layer, and the pedagogical intent flow will speak one consistent language about a student session. What the engine detects, the dashboard shows. What the professor intends, the case reflects. What the student reasons through, the analytics honor — whether or not the student named the framework. No more subsystems that disagree with each other. That is what makes this a milestone.
0.3 What this packet does NOT promise
It does not guarantee that every generated case is pedagogically perfect. The system is a constrained and validated LLM pipeline, not a deterministic authoring engine. Quality gates make output reliable, not flawless.
It does not include the institutional or admin dashboard. Separate packet.
It does not include the grading layer. Separate packet.
It does not include multi-case management, multi-language beyond Spanish and English, or student-facing post-simulation analytics beyond the current results page.

Section 1 — What This Milestone Means
1.1 The three halves that must agree
Academium has three functionally distinct halves. Until this milestone, they have developed in parallel but have not been forced to agree with each other. Making them agree is what this packet delivers.
1.1.1 Case creation
A professor turning teaching intent into a runnable simulation. Strong case creation means intent-driven, theory-anchored, decision-designed, professor-reviewable, language-consistent. T-002 shows this half is the most advanced — the framework field exists, the case is visibly theory-shaped, edit-before-publish works. What remains: pedagogical intent as a structured input, per-decision academic dimensions, and framework canonicalization.
1.1.2 Reasoning environment
The live simulation a student runs and the calibrated engine behind it. Strong reasoning environment means framework-rewarding, faithful to student voice, observable, and dashboard-truthful. T-002 shows this half runs reliably — the engine preserves logic across turns, KPIs move credibly, the reflection stage is reached cleanly. What remains: the narrator fidelity rule (still inflates), the framework-responsive narrator, the dimension-weighted signal extraction.
1.1.3 Analytics and dashboard
Everything the professor sees about what happened. Strong analytics means debrief-ready, observable, framework-accurate, and truthful in near-real time. T-002A shows this half is where the biggest gaps remain — the completion sync is still broken, framework detection is fundamentally unreliable, and the analytics layer does not trust the same signals the engine extracted. This is the half most dependent on cohesion with the other two.
1.2 Why cohesion is the central theme of v3.0
T-002A surfaced an internal contradiction that is the most important finding in any test round so far: the engine's signal extractor correctly identified strategic reasoning and tradeoff awareness throughout the student's session. The framework detector, working from the same student input, reported that Porter's Generic Strategies was not evidenced anywhere. The two subsystems disagree because they are loosely coupled — the framework detector is running keyword and signal-pattern matching independently, while the signal extractor is doing semantic extraction. The dashboard then renders the weaker output without reconciling with the stronger one.
This is not a bug in either subsystem individually. It is a failure of cohesion between them. Section 12 of this packet defines the Cohesion Architecture that binds them, and Section 4 implements the specific fix for framework detection that follows from it.
1.3 The non-negotiables
Mentorship posture — calm senior colleague voice. No praise, no judgment, no answers.
Safety posture — redirect profanity or off-topic without shame. Student text preserved verbatim always.
Realism posture — decisions are irreversible once submitted.
Revision posture — student text never deleted, cleared, or overwritten on NUDGE or BLOCK.
Tradeoff posture — every decision involves genuine trade-offs. No obviously correct option.
Professor authority — the professor can always remove, override, or replace anything the system generated or inferred.
Dashboard as surface — the dashboard reads session state. It never invents a new assessment, only aggregates what the engine produced.
Subsystem agreement — engine subsystems must agree with each other before any dashboard element renders. When they disagree, the dashboard flags the disagreement. This is new in v3.0.
1.4 What cohesion looks like in practice
A professor teaches Week 4 of Competitive Strategy. They tell Academium: teaching goal is applying Porter's Generic Strategies to a differentiation-versus-focus decision, target framework is Porter, reasoning constraint is limited budget. The system generates a case in which Decision 1 forces a strategic-positioning choice, Decision 2 forces a tradeoff between reach and specialization, and Decision 3 forces metric selection under budget pressure.
A student submits short, realistic answers using words like niche, inventory reduction, core audience — without ever saying the word Porter, cost leadership, or differentiation. The engine extracts signals showing strong strategic intent and present tradeoff awareness. The framework detector, running the semantic check defined in Section 4, recognizes that the student is reasoning through focus-strategy logic even without naming it, and returns Porter as implicit with medium confidence. The narrator, receiving this detection result, produces a consequence narrative where stakeholders respond to a clearly-positioned choice rather than a confused one.
The professor dashboard displays: a completion count that updates within 60 seconds, a framework-application table that shows Porter implicit on Decisions 1, 2, and 3 with confidence medium, a debrief question grounded in the student's actual phrasing, and a class-level module health card that shows Porter transferring at 70%. The professor walks into class tomorrow with a real, specific, defensible teaching artifact. That is the milestone.

Section 4 — Priority 2: Semantic Framework Detection
This is the single most important addition in v3.0. It addresses the central new finding from T-002A: the framework detector marks frameworks as Not Evidenced even when the student has clearly applied the underlying concept. This section defines the new detector architecture that fixes it.
4.1 The core principle
A framework is applied when the student's reasoning follows its analytical pattern — whether or not they name it, whether or not they use canonical vocabulary. The detector's job is to recognize the concept in action, not to match strings. Keyword matching is a shortcut for high-confidence explicit detection only. Every other detection path must be semantic.
4.2 The three detection tiers
Tier
Method
Rule
Explicit
Name match OR strong keyword match
Student uses the framework name OR vocabulary that is distinctive to the framework (multi-word or canonical-only terms). Keyword-based. Fast. High-confidence.
Implicit (semantic)
LLM conceptual analysis
Student's reasoning demonstrates the framework's analytical pattern without naming it. The LLM evaluates reasoning structure, not vocabulary. This is the new path. PRIMARY path for T-002 style cases.
Signal-pattern
Heuristic fallback
Existing allSignalsMet + keyword path. Used only when the semantic LLM check is unavailable. Tagged confidence: low.
4.3 Detection algorithm
In server/agents/frameworkDetector.ts, rewrite detectFrameworks to follow this flow per framework:
async function detectOneFramework(studentInput, signals, framework, language) {
// Tier 1 — Explicit
const nameMatch = regexNameMatch(framework.name, studentInput);
const strongKw = strongKeywordMatch(framework.domainKeywords, studentInput);
if (nameMatch || strongKw) {
return {
level: "explicit",
confidence: "high",
evidence_snippet: extractSnippet(studentInput, matched),
detection_method: "keyword",
reasoning: "Named the framework or used canonical vocabulary."
};
}
// Tier 2 — Semantic Implicit (PRIMARY path)
const semantic = await semanticFrameworkCheck({
studentText: studentInput,
signals,
framework,
language
});
if (semantic.applied) {
return {
level: "implicit",
confidence: semantic.confidence,
evidence_snippet: semantic.quotedReasoning,
detection_method: "semantic",
reasoning: semantic.explanation
};
}
// Tier 3 — Signal-pattern fallback (unchanged, low-confidence)
if (framework.signalPattern && signalPatternMatches(framework, signals)) {
return {
level: "implicit",
confidence: "low",
evidence_snippet: extractStudentQuote(studentInput),
detection_method: "signal_pattern",
reasoning: "Signal pattern matches."
};
}
return {
level: "not_evidenced",
confidence: "high",
evidence_snippet: null,
detection_method: "none",
reasoning: "No conceptual or keyword evidence detected."
};
}
4.4 The semantic check
semanticFrameworkCheck is a new LLM call. Input: student text, extracted signals, framework definition. Output: applied / not, confidence, quoted reasoning from student, one-sentence explanation.
4.4.1 Prompt structure
System prompt (bilingual, Es and En variants):
You are analyzing whether a student's business reasoning shows
application of a specific analytical framework, even when they
do not name it.
Framework name: <name>
Framework core concepts: <coreConcepts>
What applying this framework looks like: <conceptualDescription>
Recognition signals: <recognitionSignals>
Student's response:
"<studentText>"
Signals extracted from the student's reasoning:
- Intent: quality <X>, evidence "<Y>"
- Justification: quality <X>, evidence "<Y>"
- Tradeoff awareness: quality <X>, evidence "<Y>"
- Stakeholder awareness: quality <X>, evidence "<Y>"
- Ethical awareness: quality <X>, evidence "<Y>"
Evaluate:
1. Did their reasoning follow this framework's analytical pattern?
2. Did they consider the dimensions this framework highlights?
3. Did they reach their conclusion through this framework's logic?
Return JSON only:
{
"applied": true | false,
"confidence": "high" | "medium" | "low",
"quotedReasoning": "exact phrase or passage from student's text",
"explanation": "one sentence on how (or why not) they applied it"
}
4.4.2 Decision rules
applied must be false unless the quoted reasoning contains actual student words (not summary).
confidence is high only when at least two of the three evaluation questions are affirmatively met.
confidence is low when the application is partial — e.g., one dimension engaged, others absent.
The explanation may not use prohibited evaluation language (correct, wrong, optimal, should have, etc.).
4.4.3 Model choice
Use gpt-4o-mini with responseFormat: json and maxTokens: 256. The call is made once per framework per turn. For a case with 3 tracked frameworks and 3 turns, that is 9 semantic checks per session. At gpt-4o-mini pricing and token count, this is negligible cost.
4.4.4 Batching for performance
If multiple frameworks are tracked on the same turn, the semantic check can be batched: one LLM call evaluating all frameworks against the same student text. This reduces latency and cost.

4.8 Per-turn detection pipeline
In director.ts, the detection flow per turn becomes:
Run classification (PASS/NUDGE/BLOCK).
Run signal extraction.
Run framework detection across all tracked frameworks. In parallel with Evaluator, Domain Expert, Narrator.
Run consistency check (Section 12.3). If signal extraction shows strong dimension X and framework detection shows not_evidenced for a framework primarily mapped to dimension X, log a disagreement event and promote the detection to implicit low-confidence as a safety net.
Proceed with the rest of the pipeline.
Section 5 — Priority 3: Framework Canonicalization and Deduplication
T-002 surfaced this: "both SWOT Analysis and a separate SWOT entry appear, which may create duplicate detection or noisy analytics if both are kept." This section defines canonical framework identity so that entering "SWOT" and entering "SWOT Analysis" resolves to the same framework.

Create a new file server/agents/frameworkRegistry.ts. It exports a canonical registry of well-known business frameworks, each with a canonicalId, a display name, and a list of aliases. Initial registry includes the frameworks most common in business education:

5.3 Resolution at entry time
When the professor types a framework name in the FrameworkEditor:
The client calls a new endpoint POST /api/scenarios/resolve-framework-name. Payload: { name: string, language: "es" | "en" }.
Server checks the registry: case-insensitive match against canonicalName and all aliases (both languages).
If match found, response contains the canonical framework with all fields (canonicalId, canonicalName in the requested language, coreConcepts, conceptualDescription, recognitionSignals, suggested domainKeywords, suggested signalPattern).
If no match, response returns canonicalId: null and the client falls back to the existing suggest-framework-keywords flow, which generates canonicalId as "custom_<hash>" and fills the semantic fields via the LLM.
Client renders a confirmation: "Matched to <canonicalName>" with an option to override if the professor intended a different framework.
5.4 Deduplication at entry and generation
5.4.1 At manual entry
In FrameworkEditor.tsx, after a framework is resolved, check whether the current scenario already has a framework with the same canonicalId. If yes, reject the addition with a toast: "This framework is already added (entered as <existing_name>)." Do not allow duplicates at the UI level.
5.4.2 At generation time
When canonicalCaseGenerator.ts produces frameworks as part of the case output, post-process the frameworks array to deduplicate by canonicalId. For duplicates, merge domainKeywords and keep the first entry's other fields.
5.4.3 Legacy data
For scenarios that already have duplicate frameworks (e.g., SWOT and SWOT Analysis as T-002 observed), add a one-time migration script that runs on server boot: resolve each framework's canonicalId, merge duplicates, keep the newest. Log before-and-after for audit.
5.5 Benefits beyond deduplication
The professor sees a clean resolved name immediately, not the raw typed input.
Analytics cross-scenario become possible: all scenarios using Porter's Generic Strategies share a canonicalId and can be aggregated.
The registry-provided conceptualDescription and recognitionSignals give the semantic detector strong, consistent data. Custom frameworks still work, but registry frameworks work better.
Common typos are absorbed. "Porter's Generic" and "Porter Generic Strategies" and "Porter" all resolve to the same framework.

Section 10 — Priority 8: Decision-Design Rules
Each decision must surface a specific academic dimension. This section defines the taxonomy, auto-assignment rules, and per-dimension design constraints. Carries forward from v2.0 Section 8.
10.1 The five academic dimensions
Dimension
Surfaces Signal
Design Requirement
analytical
justification (C1)
Must require reasoning from data or from a causal chain. Options or prompt must force the student to explain why.
strategic
intent (C2)
Must require prioritization across multiple defensible paths. Long-horizon consequences implied.
stakeholder
stakeholderAwareness (C3)
Must affect at least two stakeholders with different interests. Options or prompt forces the student to acknowledge at least one.
ethical

tradeoff

Must have named cost and named benefit pulling in opposite directions. tradeoffSignature populated.
10.2 Default auto-assignment
3-decision case: [stakeholder, tradeoff, strategic].
4-decision case: [analytical, stakeholder, tradeoff, strategic].
5-decision case: [analytical, stakeholder, ethical, tradeoff, strategic].

Never assign the same primary dimension to consecutive decisions unless the professor explicitly specifies it or targetCompetencies has only one value.
Decision 1 is the entry point. For MCQ Decision 1, the dimension must be observable in the option set.

Analytical
At least one numeric data point or causal claim in the prompt.
Student invited to explain reasoning.
focusCue: "What does the data actually tell you?"
Strategic
At least two defensible paths, ideally three.
Second-order consequences implied.
focusCue: "How does this choice shape the path ahead?"
Stakeholder
At least two named stakeholders with distinct interests (must exist in scenario.initialState.stakeholders).
Student invited to consider impact on at least one named party.
focusCue: "Who is affected and how differently?"
Ethical
Tension between two legitimate goods, not good vs evil.
No moralizing in prompt itself.
focusCue: "What values are in tension here?"
Tradeoff

focusCue: "What are you willing to give up to get what?"

Section 11 — Priority 9: Reasoning Environment Calibration
With intent-driven generation and semantic framework detection, the engine itself needs to reward framework application. This section defines calibration changes to narrator, signal extractor, debrief generator, summary, and module health.
11.1 Narrator — Framework-Responsive Consequences
Current behavior
narrator.ts generates consequence narratives based on student input and RDS band. It does not differentiate between students who applied the targeted framework and students who winged it.
Target behavior
The narrator receives framework detection results. Narrative composition differs by detection state:
Explicit or implicit with confidence ≥ medium: narrator produces coherent consequence chains. Stakeholder reactions plausibly connect to the applied reasoning. The system acknowledges the thinking without naming the framework (never names it to avoid telegraphing).
Not evidenced or implicit with confidence low: narrator still produces a coherent consequence, but includes one piece of information that surfaces the dimension the student did not engage with. Teaching through consequence, not lecturing.
Implementation
In director.ts, pass frameworkDetections and decisionPoint.primaryDimension to generateNarrative.
In narrator.ts, add frameworkResponseDirective block with three variants based on detection state.
Never punish, shame, or correct. Difference is in what information the consequence surfaces, not in tone.
11.2 Signal Extractor — Dimension-Weighted
When decision has a known primaryDimension, apply modest scoring boost to the corresponding signal for that turn. Mapping: analytical→justification, strategic→intent, stakeholder→stakeholderAwareness, ethical→ethicalAwareness, tradeoff→tradeoffAwareness.
Rule: if raw extracted quality is 1 (WEAK), check for marginal evidence (targetFrameworks domainKeywords, stakeholder references for stakeholder decisions, etc.). If marginal evidence exists, promote to 2 (PRESENT). Never promote above 2. Non-mapped signals scored unchanged.

Section 12 — The Cohesion Architecture
This section defines how the engine, the dashboard, the data layer, and the pedagogical intent flow agree with each other. The central T-002A finding is that they currently do not — the signal extractor says the student reasoned strategically, the framework detector says no framework was evidenced, and the dashboard renders the contradiction. This section closes that gap.
12.1 The single data contract
There is exactly one data contract between the engine and the dashboard: the session state object as defined in shared/schema.ts. The engine writes. The dashboard reads. Nothing else communicates. No side channels. No cached facts. No implicit assumptions about state.
Every dashboard element traces to a named session state field. This was stated in v1.0. It is restated here because v3.0 enforces it structurally through the compatibility manifest (Section 12.6).
12.2 The chain of reasoning
Every piece of professor-visible insight traces through this chain:
Professor pedagogical intent
↓
Case generation (intent-driven prompt)
↓
Generated case (decisions tagged with primaryDimension, frameworks with semantic fields)
↓
Student turn submitted
↓
Classification (PASS / NUDGE / BLOCK)
↓
Signal extraction (weighted by primaryDimension)
↓
Framework detection (semantic, using framework.conceptualDescription)
↓
Consistency check (Section 12.3)
↓
Narrator + Evaluator + Domain Expert (receive detection result)
↓
Causal explainer (dashboard_reasoning_link)
↓
Session state write
↓
Cache invalidation on completion
↓
Dashboard read → aggregate → display
Every arrow in this chain is a defined interface. When any subsystem is updated, its output contract is enforced at the boundary so the next subsystem cannot silently consume malformed data.
12.3 The consistency check
After signal extraction and framework detection run for a turn, a new consistency check reconciles them. Add in director.ts, between the parallel agent calls and the result assembly.
async function checkConsistency(
signals: SignalExtractionResult,
detections: FrameworkDetection[],
frameworks: CaseFramework[],
decisionPrimaryDimension: AcademicDimension | undefined
): Promise<FrameworkDetection[]>
Rules
For each framework, if the signal mapped to its primary domain (per DIMENSION_TO_SIGNAL from Appendix C) is at PRESENT or STRONG, AND the detection returned not_evidenced, log a disagreement event via storage.createTurnEvent.
If the decision's primaryDimension maps to a signal that is PRESENT or STRONG, AND the framework most closely associated with that dimension is not_evidenced, promote that framework's detection to implicit with confidence: low. Add reasoning: "Signal extraction indicated <dimension> reasoning at <level>; framework promoted as safety net."
If multiple frameworks map to the same dimension and signals indicate reasoning in that dimension, promote only the one flagged in pedagogicalIntent.targetFrameworks. If none targeted, promote none (no false positives).
Never demote. The check only promotes detections, never downgrades them.
The goal is that a student who clearly reasons strategically never sees all their tracked frameworks marked Not Evidenced.
12.4 Subsystem output contracts
Each subsystem's output is validated at the boundary before the next consumes it. Invalid outputs trigger regeneration or fallback, never silent acceptance.

12.5 The professor-facing explainability guarantee
For every piece of information the professor sees on the dashboard, the system can answer four questions:
Where did this come from? — the exact session state field.
How was it computed? — the engine rule that wrote it.
What inputs produced it? — student text, signals, framework, or all three.
How confident are we? — high / medium / low, tied to detection_method.
This is the professor-facing guarantee. When a professor or dean asks "how did the AI decide Porter was applied implicitly here?" the answer is: "Semantic framework detection evaluated the student's reasoning on this turn (session_state.turns[1].framework_detections[0]) with confidence medium. The quoted student reasoning that triggered detection is <quote>. The framework's conceptual description the check matched against is <description>." Not a black box. Fully explainable.
12.6 Compatibility manifest
A compatibility manifest file lives in the repo. It lists every dashboard element, the endpoint that feeds it, the session state field the endpoint reads, the engine rule that writes the field, and the expected type.
The manifest validates on every CI build. When any engine change modifies a field the dashboard consumes, the build fails until the manifest is updated. This prevents silent drift.
First draft of the manifest is a deliverable of Phase 1. Maintained going forward.
12.7 Cohesion in practice — reading one dashboard value
To illustrate: when the professor sees "Porter: Implicit — Medium confidence — applied on Turn 2" in the Module Health card, the chain is:
UI element: ScenarioDashboard.tsx ModuleHealthCard.
Data source: GET /api/scenarios/:id/module-health response.frameworks[fwIdx].
Endpoint reads: session_state.turns[*].framework_detections[fwIdx].
Field written by: frameworkDetector.detectOneFramework, semantic path.
Input: session_state.turns[1].student_input + session_state.turns[1].signals_detected + scenario.initialState.frameworks[fwIdx].
Engine rule: LLM semantic check prompt defined in Section 4.4.1 with the framework's conceptualDescription.
Confidence source: the semantic check's confidence output. Explanation source: the semantic check's explanation output, echoed into FrameworkDetection.reasoning.
Any of the seven links can be inspected. None of them are hidden. That is cohesion.

/**

* Distinct from case content. Intent is the INPUT driving case
* generation, framework inference, and runtime calibration.
* Stored on scenarios.pedagogical_intent. Editable only when
* scenario has zero sessions.

export interface PedagogicalIntent {
teachingGoal: string;
targetFrameworks: Array<{
canonicalId: string | null;
name: string;
}>;
targetCompetencies: Array<"C1"|"C2"|"C3"|"C4"|"C5">;
decisionDimensions?: Array<{
decisionNumber: number;
primaryDimension: AcademicDimension;
secondaryDimension?: AcademicDimension;
}>;
courseContext?: string;
reasoningConstraint?: string;
}

| "analytical" | "strategic" | "stakeholder"
| "ethical" | "tradeoff";

};

};

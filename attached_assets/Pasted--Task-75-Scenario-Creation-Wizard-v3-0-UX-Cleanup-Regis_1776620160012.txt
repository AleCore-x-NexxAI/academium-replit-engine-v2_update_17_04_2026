# Task #75 — Scenario Creation Wizard v3.0 UX Cleanup + Registry Expansion

## What & Why

Post-v3.0, the scenario creation wizard (client/src/components/CanonicalCaseCreator.tsx) stacks the new Teaching Intent card on top of the legacy pre-v3.0 fields without deleting or consolidating them. The result is a confusing duplicated form where the professor fills out Teaching Intent (target frameworks, target competencies, per-decision dimensions) and then sees a second redundant set of fields below: setupFrameworks (duplicates intent target frameworks), tradeoffFocus (duplicated by C5 in targetCompetencies), scenarioObjective (duplicated by teachingGoal), and a separate Discipline selector that's redundant given the framework picker can carry discipline information.

Additionally:
- Language toggle leaks — Teaching Intent labels stay Spanish when UI flips to English (and vice versa).
- Framework input is a bare text field — typing "Porter" fires a destructive toast instead of inline disambiguation. Typing "Porter Five Forces" resolves to `porter_generic_strategies` instead of `porter_five_forces` (resolver bug).
- Per-decision dimension selectors show two unlabeled dropdowns side-by-side with no indication which is primary vs secondary, and no help text explaining what any dimension means.
- Registry is limited to 13 strategy-focused frameworks — unusable for marketing, finance, operations, HR courses.

This task fixes all of the above in one coordinated pass.

## Done looks like

1. The wizard, when Teaching Intent is filled, shows ONLY these fields outside the Teaching Intent card: Topic (required), Level (Pregrado/Posgrado/Ejecutivo), Language, Step count. The following fields are DELETED entirely: `setupFrameworks`, `tradeoffFocus`, `scenarioObjective`, `discipline` selector.
2. Teaching Intent is required — Generate button stays disabled until `teachingGoal.trim().length > 0`.
3. Language toggle fully switches all strings inside Teaching Intent card. No Spanish labels remain in English mode. No English labels remain in Spanish mode. Run a grep audit on the Teaching Intent card JSX and verify every user-facing string is either `language === "en" ? "..." : "..."` or uses a `t()` key with both `en` and `es` entries in `translations.ts`.
4. Framework picker redesigned as a grouped accordion with 6 discipline sections: Business, Marketing, Finance, Operations, Human Resources, Strategy. Each section is collapsible. Each has a checkbox on the group header itself (to select the discipline as a whole without picking a specific framework) AND individual framework checkboxes inside the group. Discipline-header checkbox and individual-framework checkboxes are independent — a user can check "Marketing" header only (no specific framework), OR check "Porter's Generic Strategies" inside Strategy only, OR check both in combination.
5. Hard cap of 3 selections total, counted across all groups (framework checkboxes AND discipline-header checkboxes both count as 1 toward the cap). When cap reached, remaining checkboxes disable with a subtle style change. Counter visible at top of the picker: "X of 3 selected".
6. When selections span 2+ different disciplines, a warning banner renders below the picker: "Cross-discipline selection — you've picked frameworks from [Business, Strategy]. The AI will generate a case that spans them, but it may not match a single course cleanly. Continue if intentional." Use `--color-background-warning` with `--color-border-warning` left accent.
7. Registry expanded from 13 to 42 frameworks. Registry entry structure extended with `disciplines: string[]` (multi-discipline array, replacing any single-discipline assumption). Every framework gets full bilingual semantic fields per the existing pattern: `canonicalName_en`, `canonicalName_es`, `aliases`, `coreConcepts_en`, `coreConcepts_es`, `conceptualDescription_en`, `conceptualDescription_es`, `recognitionSignals_en`, `recognitionSignals_es`, `primaryDimension`, `suggestedDomainKeywords_en`, `suggestedDomainKeywords_es`, `suggestedSignalPattern` (optional), `disciplines` (new array).
8. `Porter Five Forces` typed into the picker search or added via resolver returns `porter_five_forces` canonicalId (never `porter_generic_strategies`). Regression test asserts this.
9. Bare `Porter` input triggers INLINE disambiguation — two buttons rendered under the input labeled "Porter's Generic Strategies" and "Porter's Five Forces", not a destructive toast. Clicking either adds that canonicalId to the selection.
10. Each framework checkbox has a HelpIcon tooltip on hover showing `conceptualDescription` (truncated to ~140 chars) and the first 3-4 `coreConcepts` as comma-separated text. Both strings pulled from the registry entry in the current UI language.
11. Per-decision dimension UI, when rendered, clearly labels the two dropdowns: left one says "Primary dimension *" (required), right one says "Secondary dimension (optional)". Above each decision's pair of dropdowns add a one-line hint: "Primary is the main reasoning this decision exercises. Secondary is a related dimension it also touches." Add a HelpIcon on each dropdown OPTION explaining what that dimension means using packet §10.1 definitions: analytical = evidence & data, strategic = priorities & direction, stakeholder = interests & reactions, ethical = obligations & fairness, tradeoff = costs & sacrifices.
12. When a framework is selected, a muted preview card renders beneath it showing `conceptualDescription` (truncated) and `coreConcepts.slice(0, 4)` as small chips. Confirms to the professor "yes this is the right framework" before generation.
13. New endpoint `GET /api/frameworks/registry?language=en|es` returns `[{canonicalId, canonicalName, aliases, disciplines, primaryDimension, conceptualDescription, coreConcepts}]` for all 42 registry entries in the requested language. Used by the picker. Cache response client-side after first fetch.
